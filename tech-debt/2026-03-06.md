## TD-20260306-01
- ID: TD-20260306-01
- Title: Auth navigation fallback hardcodes `/oauth2/start` edge route
- Context: While wiring the UI auth-recovery and switch-user flows, we kept a fallback in the auth navigation helper so the app can still build auth URLs when `VITE_AUTH_START_URL` is absent.
- Location: `ui/src/utils/authNavigation.js`, consumers in `ui/src/components/AuthErrorPage/AuthErrorPage.jsx` and `ui/src/App.jsx`.
- Impact/Risk: The UI still embeds a concrete edge-auth path assumption in code. Deployments with a different auth gateway path or host can silently redirect users to the wrong location when the environment variable is missing or misconfigured.
- Why Deferred Now: Current scope was fixing auth-recovery behavior and user switching, not redesigning the frontend auth configuration contract or removing the local fallback path entirely.
- Proposed Fix: Replace the hardcoded fallback with an explicit configuration contract, or derive the auth entrypoint from a single runtime source that can be validated at startup/build time. Consider failing fast when the auth entrypoint is required but missing.
- Acceptance Signal: Auth navigation no longer contains a hardcoded `/oauth2/start` fallback; startup/build validation or a single canonical config source guarantees correct auth-entry URL generation across local and deployed environments.
- Status: open
