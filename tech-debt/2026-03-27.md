# Technical Debt - 2026-03-27

Purpose: track deferred technical debt discovered during document/revision workflow review follow-up.

## TD-20260327-01
- Status: open
- Title: Add a guardrail against workflow logic leaking back into generic update payloads
- Context: Review of the dedicated revision lifecycle endpoints confirmed that `POST /api/v1/documents/revisions/{rev_id}/overview-transition` and `POST /api/v1/documents/revisions/{rev_id}/supersede` keep progression logic narrow and explicit. The current implementation already rejects workflow-managed fields in generic update flows, but there is no focused regression guard that fails if future changes broaden `DocUpdate` or `DocRevisionUpdate` to accept workflow-controlled fields again.
- Location:
  - `api/schemas/documents.py`
  - `api/routers/documents.py`
  - `tests/api/api/test_documents_revisions_endpoints.py`
  - `documentation/api_interfaces.md`
  - `documentation/api_db_rules.md`
- Impact/Risk: A future refactor could accidentally reintroduce workflow/business logic into generic document or revision update payloads, allowing clients to bypass the dedicated lifecycle endpoints. That would weaken the DB-as-policy-engine contract, make API semantics less predictable, and increase the risk of business rules being duplicated or inconsistently enforced across endpoints.
- Why Deferred Now: The current task was review hardening and targeted regression coverage for existing lifecycle behavior, not adding a new policy-style guard suite. The current implementation is compliant, so this is a preventative follow-up rather than an active defect.
- Proposed Fix: Add a narrow contract test or lint-style assertion that explicitly fails if generic update schemas or handlers start accepting workflow-controlled fields such as revision progression controls, `rev_code_id` after creation, or workflow-managed document pointers. Keep the dedicated overview-transition and supersede endpoints as the only supported public progression paths.
- Acceptance Signal: Automated verification fails if workflow-controlled fields are reintroduced into generic document/revision update contracts, and the API/documentation still describe dedicated lifecycle endpoints as the only supported progression mechanisms.
