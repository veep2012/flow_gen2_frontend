# Technical Debt - 2026-03-07
Purpose: Track implementation-ready technical debt items identified on this date.

## TD-20260307-01
- ID: TD-20260307-01
- Title: Repository lacks a dedicated compose profile for full-environment auth/runtime validation
- Context: While implementing and validating API JWT verification, nginx bearer-token passthrough, and local Keycloak audience defaults, we confirmed that the repository can test units, API smoke flows, and local compose manually, but it does not provide one dedicated compose-based test configuration for full-environment validation.
- Location: `ci/docker-compose.yml`, `Makefile`, `ci/nginx/default.conf`, `ci/keycloak/flow-local-realm.json`, auth-sensitive API smoke flows under `tests/api/`.
- Impact/Risk: End-to-end regressions across nginx, oauth2-proxy, Keycloak, API JWT verification, and compose wiring can survive until manual testing. This increases the chance of environment-specific auth failures, issuer/audience mismatches, or ingress-path drift that unit/API-only tests do not catch.
- Why Deferred Now: Current scope was implementing Phase 3 JWT auth and local ingress behavior, not designing and stabilizing a separate compose test topology plus its automation contract.
- Proposed Fix: Add a dedicated compose test configuration or profile that brings up the full auth stack specifically for verification runs, including deterministic Keycloak realm import, nginx ingress behavior, oauth2-proxy cookie flow, bearer-token passthrough, and scripted smoke checks. Wire it into a dedicated make target so the environment can be exercised reproducibly in local development and CI.
- Acceptance Signal: A single documented command starts the dedicated compose test environment and runs deterministic smoke checks that validate cookie-based ingress, bearer-token ingress, and current-user/auth-sensitive API behavior end to end through nginx.
- Status: open

## TD-20260307-02
- ID: TD-20260307-02
- Title: Raw `X-User-Id` request header remains a production-capable identity source
- Context: During a security review of the auth path, the request identity resolver was checked to confirm whether only verified JWTs and trusted edge headers can establish user context in deployed environments.
- Location: `api/utils/database.py`
- Impact/Risk: `_set_app_user()` still accepts caller-supplied `X-User-Id` whenever `Authorization` and the trusted header are absent. If the API is ever reachable directly, or if an ingress/proxy fails to strip or overwrite that header, a client can impersonate any resolvable internal user acronym and inherit that user's database-backed permissions and RLS scope.
- Why Deferred Now: Fixing this safely requires a deployment-contract change across FastAPI, nginx/oauth2-proxy forwarding rules, local developer workflows, and the auth/API documentation rather than a review-only update.
- Proposed Fix: Gate `X-User-Id` support behind an explicit non-production flag or remove it entirely from production-capable paths. Prefer verified bearer JWT or trusted edge headers only, and add automated tests that prove spoofed `X-User-Id` is rejected in deployed-mode configuration.
- Acceptance Signal: In production-mode configuration, requests that carry only `X-User-Id` return `401`, while JWT/trusted-header flows still work and deployment docs explicitly state that end-user identity cannot be established from a raw client header.
- Status: open

## TD-20260307-03
- ID: TD-20260307-03
- Title: People directory and permission listings are readable without authentication
- Context: During the repository-wide security review, router-level auth coverage was compared across API modules to identify endpoints that bypass the standard `require_effective_identity` dependency.
- Location: `api/routers/people.py`, `tests/api/unit/test_auth_router_matrix.py`
- Impact/Risk: The `people` router is mounted without a router-level auth dependency, while list endpoints such as `/api/v1/people/roles`, `/api/v1/people/persons`, `/api/v1/people/users`, and `/api/v1/people/permissions` are exposed as plain `GET` routes. This allows anonymous callers to enumerate internal users, roles, persons, and authorization mappings, materially lowering the cost of reconnaissance and targeted abuse.
- Why Deferred Now: Tightening these endpoints needs an API contract decision about which directory data must stay public versus authenticated, followed by coordinated test and documentation updates. That is beyond a review-only task.
- Proposed Fix: Add `require_effective_identity` at router or endpoint level for non-public people endpoints, then explicitly whitelist only the minimal read surfaces that are intended to be public. Extend the auth router matrix to cover the resulting policy.
- Acceptance Signal: Anonymous requests to non-public people endpoints return `401`, the auth router matrix includes those routes, and documentation distinguishes any intentionally public directory endpoints from authenticated ones.
- Status: open

## TD-20260307-04
- ID: TD-20260307-04
- Title: Lookup catalog endpoints are exposed anonymously
- Context: The same router auth review checked shared reference-data surfaces for parity with the authenticated document, file, notification, and distribution-list routers.
- Location: `api/routers/lookups.py`, `tests/api/unit/test_auth_router_matrix.py`
- Impact/Risk: `/api/v1/lookups/*` list endpoints are mounted without `require_effective_identity`, exposing areas, disciplines, projects, units, jobpacks, document revision statuses, and UI behavior mappings to any caller. This leaks internal workflow metadata and status topology that should typically be available only to authenticated users or explicitly public clients.
- Why Deferred Now: Applying auth here requires agreement on whether any lookup endpoints must remain anonymously readable for bootstrapping UI flows, plus corresponding scenario/test/doc updates.
- Proposed Fix: Require effective identity for lookup endpoints by default, then carve out narrowly scoped exceptions only where anonymous bootstrap is genuinely required. Add tests to prevent silent regression back to public access.
- Acceptance Signal: Unauthenticated requests to protected lookup endpoints return `401`, allowed public exceptions are documented, and auth coverage tests include the lookup router.
- Status: open

## TD-20260307-05
- ID: TD-20260307-05
- Title: Prometheus-style auth telemetry is publicly reachable at `/metrics`
- Context: The security review inspected system endpoints to verify whether observability surfaces were protected at the same level as the underlying authentication and authorization mechanisms they describe.
- Location: `api/routers/system.py`, `documentation/api_interfaces.md`
- Impact/Risk: `/metrics` is unauthenticated and returns auth-related counters such as JWT validation failures, identity parse failures, and current-user resolution failures. Even without secrets, this gives unauthenticated callers operational insight into authentication behavior, failure modes, and endpoint activity that is useful for reconnaissance.
- Why Deferred Now: Restricting metrics access can affect existing scraping, local smoke tests, and deployment assumptions, so it needs a coordinated operational change instead of an isolated review tweak.
- Proposed Fix: Protect `/metrics` behind trusted infrastructure only, such as network-level restriction, a dedicated scrape token, or authenticated internal ingress. Keep any public health signal limited to `/health`.
- Acceptance Signal: External unauthenticated requests cannot read `/metrics`, internal monitoring still scrapes it successfully, and tests/docs reflect the new access contract.
- Status: open

## TD-20260307-06
- ID: TD-20260307-06
- Title: API CORS defaults are permissive enough for unintended cross-origin token use
- Context: The security review checked default edge/browser exposure for API routes, especially after direct bearer-token support was added to the backend.
- Location: `api/main.py`
- Impact/Risk: The API defaults `ALLOWED_ORIGINS` to `*` and allows all methods and headers. While credentials are disabled under the wildcard, this still permits any origin to issue browser-based cross-origin requests that include bearer tokens under direct-token usage. That broadens the blast radius of token mishandling and makes origin-based containment weaker than expected.
- Why Deferred Now: Tightening CORS requires a deployment-aware origin allowlist, coordinated UI/nginx configuration, and test updates for local development and integration environments.
- Proposed Fix: Replace the wildcard default with an explicit origin allowlist per environment, keep methods/headers narrow where possible, and document local-development overrides separately from deployed defaults.
- Acceptance Signal: Production-capable defaults reject unexpected origins, approved UI origins continue to function, and deployment docs define the origin allowlist expected for each environment.
- Status: open
