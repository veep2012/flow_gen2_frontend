# Technical Debt - 2026-03-13

Purpose: track deferred technical debt discovered during API authorization review work on commented-file ownership enforcement.

## TD-20260313-03
- Status: resolved
- Title: Define application-level security rules above technical enforcement layers
- Context: Recent authorization reviews exposed that the system has strong technical enforcement pieces (`require_effective_identity`, workflow functions, RLS, grants), but lacks one authoritative application-level policy that states who is allowed to perform which business actions and when delegation/on-behalf-of behavior is allowed.
- Location:
  - `documentation/authorization_rls_matrix.md`
  - `documentation/authentication_rls_matrix_as_is.md`
  - `documentation/api_db_rules.md`
  - `api/routers/`
  - `ci/init/07_workflow.sql`
- Impact/Risk: Without a business-level authorization model, individual endpoints and workflow functions can remain internally consistent yet still encode the wrong product policy. That leads to piecemeal decisions about who may create documents, create revisions, upload files, replace or drop artifacts, administer notifications, or act on behalf of other users. The result is authorization drift, review ambiguity, and a higher chance of silent privilege expansion over time.
- Why Deferred Now: This is broader than a single endpoint fix. It requires product, backend, and database alignment on role/action semantics before code changes can be applied safely.
- Proposed Fix: Create a dedicated application-level security specification that defines action-by-action business authorization rules for core resources (`doc`, `doc_revision`, `files`, `files_commented`, `written_comments`, `notifications`, `distribution_list`). It should explicitly cover:
  - who can create, edit, replace, drop, cancel, delete, and read each resource
  - which actions are owner-only, role-based, scope-based, or superuser/admin-only
  - whether acting on behalf of another user is allowed, and under what audited conditions
  - how application-level rules map onto DB workflow functions, API contracts, and tests
- Acceptance Signal: The repository has an approved application-level authorization document and matching enforcement/tests, and endpoint reviews can validate behavior against a single source of truth instead of inferring policy from scattered implementation details.
- Resolution: On 2026-03-18 the repository added `documentation/application_authorization_policy.md` as the authoritative business-level authorization policy and linked existing auth/RLS enforcement docs to it. This closes the documentation gap by giving reviews one source of truth for actor, scope, ownership, and on-behalf rules.

## TD-20260313-04
- Status: open
- Title: Commented-file replace can orphan old MinIO objects after post-commit cleanup failure
- Context: Review of the new commented-file replace flow confirmed that the endpoint uploads the new object, commits the DB update, and only then performs best-effort deletion of the superseded MinIO object.
- Location:
  - `api/routers/files_commented.py`
- Impact/Risk: If old-object deletion fails after the DB commit succeeds, the user-visible state remains correct but the previous object is left behind in MinIO with no durable recovery path. Repeated failures can accumulate orphaned blobs and increase storage drift/cost over time.
- Why Deferred Now: The current task scope was review and debt capture. A robust fix needs an operational design decision around retries, background cleanup, and observability rather than a small local patch.
- Proposed Fix: Add durable observability and recovery for post-commit cleanup failures. Minimum acceptable improvement is structured warning/metrics for old-object cleanup failure. Preferred follow-up is a persisted cleanup queue or periodic orphan-reconciliation job that can remove superseded commented-file objects safely.
- Acceptance Signal: Failed superseded-object cleanup events are observable and recoverable, and the system has an automated path to reconcile orphaned commented-file objects instead of relying on ad hoc log inspection.

## TD-20260313-05
- Status: open
- Title: Compose cookie-login smoke test depends on brittle Keycloak HTML scraping
- Context: Review of `make test-compose` found that the cookie-auth smoke path drives the Keycloak login page by fetching protected content, scraping the login form action from returned HTML, and submitting credentials through that parsed action URL.
- Location:
  - `scripts/test-compose.sh`
  - `documentation/test_scenarios/compose_auth_smoke_test_scenarios.md`
- Impact/Risk: The check is practical for end-to-end ingress smoke coverage, but it is tightly coupled to current Keycloak HTML/template structure through a regex match on `login-actions/authenticate`. Keycloak template or version changes can break the test even when auth behavior is still fundamentally correct, and current diagnostics do not expose much context beyond “Could not locate Keycloak login form action”.
- Why Deferred Now: The current task scope was review and debt capture. Improving this test would require choosing between keeping a brittle-but-simple smoke approach, switching to a more resilient HTML/form parsing helper, or redesigning the cookie-session validation path.
- Proposed Fix: Isolate the cookie-login scraping into a helper with clearer diagnostics, including final URL and basic page/title context on failure. Document explicitly that this is a smoke test coupled to Keycloak UI shape and may need adjustment during Keycloak upgrades. If feasible, replace raw regex scraping with a narrower parser that targets form semantics more robustly.
- Acceptance Signal: When Keycloak login markup changes, `make test-compose` failures clearly identify the failing page and redirect target, and the cookie-auth smoke logic is documented as intentionally UI-coupled or updated to a more resilient parsing strategy.

## TD-20260313-06
- Status: open
- Title: Notification inbox listing lacks explicit coverage for fallback and nonexistent-recipient semantics
- Context: Review of `list_notifications_for_recipient()` confirmed that current tests cover explicit recipient queries and missing-identity failure, but do not pin the remaining contract edges introduced by the current fallback behavior.
- Location:
  - `api/routers/notifications.py`
  - `tests/api/api/test_notifications_endpoints.py`
  - `documentation/test_scenarios/notifications_api_test_plan.md`
- Impact/Risk: The list endpoint currently uses explicit `recipient_user_id` when provided and falls back to the effective current user otherwise. Without direct tests for fallback and nonexistent-recipient cases, future refactors can silently change behavior or error semantics without detection. That is especially risky while the endpoint still carries the broader cross-user override design debt.
- Why Deferred Now: The current task scope was review and debt capture. Tightening this properly should be done alongside the final product decision for notification inbox authorization semantics.
- Proposed Fix: Add scenario and automated coverage for:
  - authenticated fallback to current user when `recipient_user_id` is omitted
  - explicit nonexistent `recipient_user_id` expected behavior
  - keep existing missing-identity `401` coverage
  - revise or remove explicit cross-user query coverage depending on the resolution of `TD-20260313-02`
- Acceptance Signal: Notification inbox tests and scenario docs explicitly lock down fallback and nonexistent-recipient behavior, so future changes to list semantics fail fast in CI instead of drifting silently.

## TD-20260313-07
- Status: open
- Title: File and commented-file creation is not yet constrained by document/revision status
- Context: Review of file-related workflows identified a missing business-rule layer around when uploads are allowed. Current create flows validate identity, references, mimetype, and uniqueness, but do not clearly enforce a status-based rule that limits file creation and commented-file creation to allowed document/revision states.
- Location:
  - `api/routers/files.py`
  - `api/routers/files_commented.py`
  - `ci/init/07_workflow.sql`
  - `documentation/document_flow.md`
  - `documentation/api_interfaces.md`
- Impact/Risk: Users may be able to create base files or commented files while a document/revision is in a status where uploads should be frozen or prohibited by business process. That can break lifecycle discipline and create ambiguity around which statuses are editable versus locked.
- Why Deferred Now: This requires an explicit product/workflow decision about allowed statuses for file and commented-file creation, then coordinated enforcement in workflow functions, API docs, and tests.
- Proposed Fix: Define the allowed status matrix for:
  - base file creation
  - base file replacement/update
  - commented-file creation
  - commented-file replacement/delete
  Then enforce it in workflow SQL functions using the authoritative revision/document status, and add matching scenario coverage and API documentation.
- Acceptance Signal: File and commented-file write actions are rejected outside the approved status set, and the allowed-status policy is explicitly documented and covered by automated tests.
