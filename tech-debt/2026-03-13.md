# Technical Debt - 2026-03-13

Purpose: track deferred technical debt discovered during API authorization review work on commented-file ownership enforcement.

## TD-20260313-01
- Status: open
- Title: Commented-file delete lacks owner-or-superuser authorization
- Context: While fixing `POST /api/v1/files/commented/{id}/replace` to enforce owner-or-superuser authorization, the adjacent delete flow was reviewed for the same actor-sensitive pattern.
- Location:
  - `api/routers/files_commented.py`
  - `ci/init/07_workflow.sql`
- Impact/Risk: `DELETE /api/v1/files/commented/{id}` currently deletes by visible `id` only. The router loads the row from `workflow.v_files_commented` and then calls `workflow.delete_file_commented(:id)` without passing the acting user. Any caller with read visibility to that commented file can therefore delete another user's commented-file payload. This is the same class of ownership/impersonation problem that existed on replace before the current fix.
- Why Deferred Now: The current task scope was limited to reviewing for similar concerns and recording any confirmed gaps. Changing delete requires coordinated API, workflow-function, security grant, tests, and documentation updates similar to the replace fix.
- Proposed Fix: Change `workflow.delete_file_commented` to accept `p_actor_user_id`, lock the row, and reject non-owner non-superuser actors. Update `api/routers/files_commented.py` to pass `get_effective_user_id(db)` and map the workflow permission error to `403` with fail-closed `404` tolerated where RLS hides the row.
- Acceptance Signal: A non-owner non-superuser cannot successfully delete another user's commented file; API smoke coverage verifies owner success and unauthorized actor failure for `DELETE /api/v1/files/commented/{id}`.

## TD-20260313-02
- Status: open
- Title: Notification inbox listing allows cross-user recipient override
- Context: While reviewing actor-sensitive notification flows, create/replace/delete/read mutations were confirmed to bind correctly to the acting user or sender. The list endpoint was then checked for equivalent trust-boundary issues.
- Location:
  - `api/routers/notifications.py`
- Impact/Risk: `GET /api/v1/notifications` accepts optional `recipient_user_id` and uses it directly when present. Because the router already has effective session identity, this lets a caller request another user's inbox by overriding the recipient in the query string. That breaks the expected “current user inbox” boundary unless cross-user inbox inspection is an intentional privileged feature.
- Why Deferred Now: The current task scope was limited to review and debt capture, not changing notification API behavior. Fixing this cleanly needs a product/API decision: either remove the override for normal flows or keep it only for a privileged admin path with explicit authorization semantics and tests.
- Proposed Fix: For the default inbox endpoint, ignore `recipient_user_id` and always use `get_effective_user_id(db)`. If cross-user inspection is truly required, split it into a separate privileged endpoint or add an explicit superuser-only authorization check plus dedicated documentation and tests.
- Acceptance Signal: A normal authenticated user cannot list another user's notification inbox through `GET /api/v1/notifications`; automated tests verify current-user-only behavior or privileged-only override behavior, depending on the chosen design.

## TD-20260313-03
- Status: open
- Title: Define application-level security rules above technical enforcement layers
- Context: Recent authorization reviews exposed that the system has strong technical enforcement pieces (`require_effective_identity`, workflow functions, RLS, grants), but lacks one authoritative application-level policy that states who is allowed to perform which business actions and when delegation/on-behalf-of behavior is allowed.
- Location:
  - `documentation/authorization_rls_matrix.md`
  - `documentation/authentication_rls_matrix_as_is.md`
  - `documentation/api_db_rules.md`
  - `api/routers/`
  - `ci/init/07_workflow.sql`
- Impact/Risk: Without a business-level authorization model, individual endpoints and workflow functions can remain internally consistent yet still encode the wrong product policy. That leads to piecemeal decisions about who may create documents, create revisions, upload files, replace or drop artifacts, administer notifications, or act on behalf of other users. The result is authorization drift, review ambiguity, and a higher chance of silent privilege expansion over time.
- Why Deferred Now: This is broader than a single endpoint fix. It requires product, backend, and database alignment on role/action semantics before code changes can be applied safely.
- Proposed Fix: Create a dedicated application-level security specification that defines action-by-action business authorization rules for core resources (`doc`, `doc_revision`, `files`, `files_commented`, `written_comments`, `notifications`, `distribution_list`). It should explicitly cover:
  - who can create, edit, replace, drop, cancel, delete, and read each resource
  - which actions are owner-only, role-based, scope-based, or superuser/admin-only
  - whether acting on behalf of another user is allowed, and under what audited conditions
  - how application-level rules map onto DB workflow functions, API contracts, and tests
- Acceptance Signal: The repository has an approved application-level authorization document and matching enforcement/tests, and endpoint reviews can validate behavior against a single source of truth instead of inferring policy from scattered implementation details.
