# Technical Debt - 2026-03-19

Purpose: track deferred technical debt discovered during revision-code lifecycle integrity review work.

## TD-20260319-03
- Status: open
- Title: `revertible` semantics are under-specified across schema, API, and tests
- Context: Review of the revision-code lifecycle model found that the rule "backward movement is allowed only when `revertible = true`" is directionally useful but still ambiguous without a precise contract. The current description does not clearly state whether revert is limited to the immediate predecessor, how that predecessor is identified, what should happen at the start step, how absent or ambiguous predecessors are handled, or whether revertibility is enforced structurally, in application logic, or both.
- Location:
  - `documentation/`
  - `api/`
  - `tests/`
  - `ci/init/07_workflow.sql`
- Impact/Risk: If `revertible` is not defined precisely, implementers and API consumers can infer different behaviors for backward transitions. That can cause drift between schema assumptions, endpoint behavior, and test expectations, especially around edge cases such as the start node, missing predecessors, or any state model where predecessor discovery is not obvious.
- Why Deferred Now: The current task scope was review and debt capture, not clarifying and propagating the full `revertible` contract through schema docs, API docs, and tests. Resolving this requires a single explicit rule and coordinated updates across multiple layers.
- Proposed Fix: Define `revertible` semantics explicitly in schema documentation, API documentation, and automated tests/examples. At minimum, specify:
  - whether revert is allowed only to the immediate predecessor
  - how the predecessor is discovered
  - expected behavior for the start step
  - expected behavior when a predecessor is absent or ambiguous
  - whether revertibility is represented and enforced structurally, in application logic, or both
  Then add tests and examples that lock those semantics down.
- Acceptance Signal: Documentation and tests define one unambiguous backward-transition rule for `revertible`, and API/database behavior matches that contract for normal and edge-case flows.

## TD-20260319-05
- Status: open
- Title: Revision-code migration and seed path needs explicit identity and foreign-key safety validation
- Context: Review of the revision-code change noted that this PR appears to evolve a reference-style table into a lifecycle-aware table. That makes migration and seed behavior a primary risk area, especially if existing environments already contain revision-code rows or if downstream tables reference `rev_code_id`.
- Location:
  - `ci/init/`
  - `migrations/`
  - `tests/`
  - `documentation/`
- Impact/Risk: Even if the target schema is correct, a weak migration path can break deployed environments through nondeterministic seed order, invalid FK transitions during inserts/updates, accidental identity remapping, or assumptions about idempotency that do not hold in repeatable setup flows. The highest-risk failure is changing `rev_code_id` identity semantics in a way that breaks downstream references without immediate visibility.
- Why Deferred Now: The current task scope was review and debt capture, not a full migration audit across existing environments, seed flows, and dependent tables. Closing this properly requires checking migration/seed scripts, downstream references, and targeted compatibility tests.
- Proposed Fix: Add migration notes and automated validation for:
  - existing environment migration cleanliness
  - deterministic seed order
  - foreign-key validity across insert/update phases
  - idempotency where repeatable setup flows expect it
  - preservation of existing `rev_code_id` identity semantics when referenced externally
  If any external tables refer to revision codes, add an explicit migration note or automated test proving those references remain intact after migration and reseeding.
- Acceptance Signal: Migration and seed tests show that existing environments upgrade cleanly, `rev_code_id` references remain valid, and repository documentation states the expected identity-preservation and idempotency guarantees.
