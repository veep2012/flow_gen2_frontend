Purpose: track deferred technical debt discovered during revision-code lifecycle integrity review work.

## TD-20260319-01
- Status: resolved
- Title: Revision-code chain invariants need explicit database-level validation coverage
- Context: Review of the revision-code lifecycle model concluded that the database invariants for cycle prevention and single ordered path semantics are likely correct in intent but remain the highest-risk part of the change set until they are validated against insert, update, and migration edge cases.
- Location:
  - `ci/init/07_workflow.sql`
  - `tests/`
  - `documentation/test_scenarios/`
- Impact/Risk: Lifecycle integrity rules are easy to describe but fragile to enforce in practice. Gaps in constraint or trigger validation can allow cycles, self-reference, multiple starts, multiple finals, invalid final-node successors, duplicate predecessors, disconnected chains, or migration-time inconsistencies to slip through. If constraint timing is misunderstood, multi-row updates or seed/migration flows can also fail unexpectedly or pass invalid intermediate states.
- Why Deferred Now: The current task scope was review and debt capture, not expanding database invariant tests or reworking constraint timing. Closing this properly requires targeted DB-level validation, test additions, and likely explicit documentation of immediate versus deferred behavior.
- Proposed Fix: Add or verify database-focused tests and migration validation for:
  - valid linear chain creation
  - inserting the first/start row
  - inserting the final row
  - updating `next_rev_code_id`
  - changing which row is start
  - changing which row is final
  - rejection of cycles
  - rejection of self-reference
  - rejection of multiple starts
  - rejection of multiple finals
  - rejection of `final = true` with a non-null `next_rev_code_id`
  - rejection of two rows pointing to the same successor, if single-predecessor semantics are intended
  - rejection of disconnected/orphan nodes, if the model intends one complete chain
  - transaction behavior during multi-row migrations/seeding
  - whether relevant constraints are immediate or deferred, with behavior documented accordingly
- Acceptance Signal: Automated tests and scenario/docs explicitly prove the allowed linear-chain lifecycle and reject invalid graph shapes and invalid transition states across inserts, updates, and migration-style multi-row transactions.
- Resolution: On 2026-03-19 the repository expanded revision-overview validation coverage in `tests/api/api/test_get_endpoints.py` and `documentation/test_scenarios/get_endpoints_api_test_scenarios.md`. The coverage now proves path-based ordering, rejects self-reference and non-self cycles, rejects invalid final/start mutations, rejects duplicate-predecessor and extra start/final inserts, and verifies a valid ordered multi-row reconfiguration can succeed inside a rolled-back transaction while start/final/predecessor uniqueness remains immediate rather than deferred. `tests/api/api/test_scenario_traceability.py` was updated so the new scenario coverage is enforced by the traceability contract.

## TD-20260319-02
- Status: open
- Title: Revision-code connectivity semantics are not explicit enough to rule out disconnected rows
- Context: Review of the revision-code lifecycle design found that the phrase "single ordered path semantics" is still ambiguous unless the repository explicitly states whether every row must belong to one connected start-to-final chain or whether disconnected acyclic islands are allowed. If only local predecessor/successor constraints are enforced, hidden disconnected nodes may still pass validation.
- Location:
  - `ci/init/07_workflow.sql`
  - `documentation/`
  - `tests/`
  - `documentation/test_scenarios/`
- Impact/Risk: A schema can reject cycles and obvious local violations while still allowing disconnected rows, unreachable nodes, or separate acyclic islands. That leaves lifecycle semantics under-specified, creates review ambiguity, and risks production data that technically satisfies local constraints but violates the intended ordered-path model.
- Why Deferred Now: The current task scope was review and debt capture, not clarifying the canonical connectivity rule across schema, docs, and tests. Resolving this requires an explicit product/technical decision and then coordinated enforcement and coverage updates.
- Proposed Fix: Make the invariant explicit in both documentation and automated tests. Choose and enforce one rule:
  - all rows must belong to the unique path from start to final
  - multiple disconnected valid chains are forbidden
  Then add tests that prove disconnected rows, unreachable nodes, and hidden acyclic islands are rejected if they are not allowed by the model.
- Acceptance Signal: The repository documentation states the required connectivity model unambiguously, and automated tests fail whenever a disconnected row or disconnected valid chain is inserted contrary to that rule.
