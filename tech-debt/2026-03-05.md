## TD-20260305-01
- ID: TD-20260305-01
- Title: Identity header precedence allows less-trusted `X-User-Id` to override trusted identity header
- Context: While validating Phase 3 trusted identity flow, we observed that backend identity resolution checks `X-User-Id` before `X-Auth-User`.
- Location: `api/utils/database.py` (`_set_app_user` header resolution order), `ci/nginx/default.conf` trusted-header forwarding.
- Impact/Risk: If backend is reachable without strict proxy stripping/rewriting, a caller can supply `X-User-Id` and bypass intended trusted-header authentication semantics. This can cause identity spoofing risk and inconsistent `400` vs `401` auth behavior.
- Why Deferred Now: Current task was advisory/testing support for Phase 3, not an auth flow refactor plus deployment/network hardening change.
- Proposed Fix: In production mode, ignore `X-User-Id` and use only `TRUSTED_IDENTITY_HEADER` (default `X-Auth-User`), or at minimum prefer trusted header over `X-User-Id` and reject requests carrying client-supplied `X-User-Id`. Enforce proxy policy to strip inbound identity headers and set only trusted identity after OIDC verification.
- Acceptance Signal: Automated tests verify trusted-header precedence and reject/ignore `X-User-Id` in trusted mode; deployment check confirms backend is not directly exposed and proxy strips identity headers before setting canonical trusted header.
- Phase 3 Blocking: yes
- Implementation Tasks:
  - Update `api/utils/database.py::_set_app_user` resolution order/policy so trusted identity path is authoritative in non-local production behavior.
  - Add/extend unit tests for precedence behavior and status semantics when both headers are present.
  - Add/extend API smoke coverage to validate trusted-header identity flow end-to-end.
  - Confirm proxy configuration strips inbound identity headers and sets canonical trusted header only.
- Resolution:
  - Backend now resolves trusted identity header (`X-Auth-User` by default) before `X-User-Id` and fails closed with `401` when trusted identity is unresolved.
  - Compose nginx now forwards canonical `X-Auth-User` to the API and clears `X-User-Id` on the API hop.
  - Automated current-user smoke coverage verifies trusted-header precedence and fail-closed behavior when both headers are present.
- Status: closed

## TD-20260305-03
- ID: TD-20260305-03
- Title: UI does not render dedicated authentication error page for `400`/`401` responses
- Context: During auth hardening follow-up, backend now returns explicit auth-related `400` and `401` outcomes, but UI fallback behavior is not standardized to a dedicated authentication error view.

- Impact/Risk: Users can see ambiguous or stale UI state after authentication failures instead of clear recovery guidance. This increases support noise and can mask real auth/session misconfiguration.
- Why Deferred Now: Current scope prioritized backend identity/session security and API contract hardening; UI auth-error state design and routing contract were intentionally postponed.
- Proposed Fix: Add a dedicated UI authentication error page/state and route all auth-related `400`/`401` responses to it with clear messaging and recovery actions (re-login, refresh session, contact support path).
- Acceptance Signal: UI behavior tests cover `400` and `401` auth responses and assert rendering of the dedicated auth-error page with deterministic fallback behavior.

  - Shared auth-aware fetch handling is wired through current-user bootstrap, document-loading flows, and auth-sensitive file/distribution-list UI calls so stale UI state is replaced deterministically.
  - UI behavior tests assert dedicated auth-error rendering for both `400` and `401` responses.
- Status: closed
