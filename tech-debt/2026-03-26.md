# Technical Debt - 2026-03-26

Purpose: track deferred technical debt discovered during document/revision workflow and frontend usability follow-up.

## TD-20260326-01
- Status: open
- Title: Frontend document workspace lacks an explicit refresh button
- Context: Review of the current document/revision workflow surfaced a usability gap in the frontend: document data can be reloaded through internal `reloadDocuments()` calls, but the UI does not expose a direct manual refresh action for users when they need to resync the document list or current workspace state on demand.
- Location:
  - `ui/src/App.jsx`
  - `ui/src/hooks/useFetchDocuments.js`
  - `ui/src/components/ProjectsPanel/ProjectsPanel.jsx`
  - `ui/src/components/DocumentTable/DocumentTable.jsx`
- Impact/Risk: Users have no obvious recovery path when data looks stale after concurrent edits, backend-side workflow changes, or intermittent fetch failures. That increases confusion around whether the system state has updated, encourages full page reloads, and makes normal refresh/retry behavior less discoverable than it should be.
- Why Deferred Now: The current task scope was backend workflow-rule adjustment, not frontend implementation. Adding the control properly should be done as a small UI task so placement, loading state, and retry behavior are defined together.
- Proposed Fix: Add a visible refresh button in the main document workspace toolbar or adjacent document navigation controls. The button should call the existing document reload flow, show disabled/loading feedback while a fetch is in progress, and avoid wiping current selection unless the refreshed dataset no longer contains the selected item.
- Acceptance Signal: Users can trigger a manual document/revision refresh from the frontend without reloading the page, and the control clearly indicates when refresh is running or complete.
